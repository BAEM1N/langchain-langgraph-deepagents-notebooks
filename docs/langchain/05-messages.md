# Messages in LangChain

## Overview

Messages form the foundational communication unit in LangChain, representing both model inputs and outputs while preserving conversation context and metadata. Each message contains three key components: a role identifier (such as `system` or `user`), content payload (text, images, audio, documents), and optional metadata fields including response details and token usage information.

## Core Message Types

### SystemMessage

These messages prime model behavior by establishing initial instructions. They define the model's role, set conversational tone, and establish response guidelines. Usage ranges from basic instructions to detailed persona specifications.

### HumanMessage

Representing user interactions, these messages support multimodal content including text, images, audio, and files. Optional metadata fields like `name` and `id` help identify users and enable message tracing.

### AIMessage

Model-generated responses containing text, tool invocations, and provider-specific metadata. The `AIMessage` class includes attributes for:

- `text`: String content
- `tool_calls`: Model-generated function invocations
- `usage_metadata`: Token counts and consumption details
- `response_metadata`: Provider-specific response information

### ToolMessage

These relay tool execution results back to models in tool-calling workflows. Required attributes include the tool result content, matching tool call identifier, and tool name. An optional `artifact` field stores supplementary data without sending it to the model.

## Message Content Formats

Messages accept content in three formats:

1. **String format**: Simple text prompts ideal for standalone requests
2. **Message objects**: Lists of typed message instances for multi-turn conversations
3. **Dictionary format**: OpenAI-compatible chat completion structure

## Multimodal Content Support

LangChain standardizes multimodal inputs across providers:

- **Images**: URL-based, base64-encoded, or provider file IDs
- **Audio files**: Supported formats with MIME type specification
- **Video files**: Base64 or file ID delivery
- **Documents**: PDF and text file support with optional metadata

Content blocks can include provider-native structures or LangChain standard blocks for type-safe processing.

## Standard Content Blocks

Messages implement a `content_blocks` property that parses content into standardized representations:

- `TextContentBlock`: Standard text output with optional annotations
- `ReasoningContentBlock`: Model reasoning and thinking steps
- `ImageContentBlock`, `AudioContentBlock`, `VideoContentBlock`, `FileContentBlock`: Multimodal data
- `ToolCall`, `ToolCallChunk`: Function invocation structures
- `ServerToolCall`, `ServerToolResult`: Server-side execution results
- `NonStandardContentBlock`: Provider-specific escape hatch for experimental features

## Practical Usage Patterns

### Basic Chat Invocation

```python
from langchain.chat_models import init_chat_model
from langchain.messages import HumanMessage, AIMessage, SystemMessage

model = init_chat_model("gpt-5-nano")
system_msg = SystemMessage("You are a helpful assistant.")
human_msg = HumanMessage("Hello, how are you?")

messages = [system_msg, human_msg]
response = model.invoke(messages)
```

### Managing Conversation History

When manually creating `AIMessage` objects for conversation history, insert them as if generated by the model to maintain proper context flow.

### Tool Calling Integration

When models invoke tools, the response contains `tool_calls` with name, arguments, and unique identifiers. Execution results return as `ToolMessage` objects, matching the original call ID.

### Token Usage Tracking

The `usage_metadata` field on `AIMessage` responses contains:

- `input_tokens`: Prompt token count
- `output_tokens`: Response token count
- `total_tokens`: Combined count
- Provider-specific breakdowns for audio, caching, or reasoning tokens

### Streaming Support

During streaming operations, models yield `AIMessageChunk` objects that combine into complete messages, enabling progressive token consumption tracking.

## Integration with Chat Models

Chat models consume message sequences and return `AIMessage` responses in stateless interactions. For conversation persistence and context management, LangChain provides built-in features for history storage and message trimming strategies to manage token budgets.

---

**Note**: The `artifact` field in `ToolMessage` serves as a programmatically-accessible storage mechanism for supplementary data like document identifiers, enabling downstream retrieval metadata access without cluttering model context.
